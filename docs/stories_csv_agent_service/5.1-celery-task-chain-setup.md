# Story 5.1: Celery Task Chain Setup

## Status

Draft

## Story

**As a** developer,
**I want** the full pipeline orchestrated as a Celery chain,
**so that** long-running analysis runs asynchronously without blocking the API.

## Acceptance Criteria

1: Celery app configured with Redis broker (`redis://redis:6379/0`)
2: 5 Celery tasks defined: `preprocess_task`, `detect_anomalies_task`, `fix_data_task`, `generate_report_task`, `export_pdf_task`
3: `PipelineService.run_full_pipeline(dataset_id, config)` creates `chain()` and `apply_async()`
4: Returns `pipeline_id` (UUID) for status tracking
5: Each task updates `pipeline_runs.current_step` when starting
6: If any task fails → pipeline status = `failed`, save `error_message`
7: Task retry: max 3 retries with exponential backoff for transient errors
8: Config object: `{auto_fix: bool, language: str, report_style: str, model_override?: str}`

### Patterns & Conventions

- **Layered architecture**: Endpoint → PipelineService → Celery Tasks → Individual Services
- **Async**: `PipelineService` methods are `async def`; Celery tasks are synchronous (run in worker process)
- **Task chaining**: Celery `chain()` passes return value of each task as first arg to next
- **Error handling**: `on_failure` callback updates pipeline status; individual task errors caught and logged
- **Config via env vars**: `CELERY_BROKER_URL`, `CELERY_RESULT_BACKEND`, `CELERY_WORKER_CONCURRENCY`
- **Pipeline errors**: Tracked in `pipeline_runs` table with `error_message` field and `failed` status

### Tech Stack

- **Celery 5.x** — task queue, `chain()`, `apply_async()`, retry with backoff
- **Redis 7 (Alpine)** — message broker (port 6379) and result backend
- **SQLAlchemy 2.x** — `pipeline_runs` model with JSONB config
- **Pydantic** — `PipelineConfig` validation
- **FastAPI** — endpoint layer (Story 5.2)
- **Alembic** — migration for `pipeline_runs` table
