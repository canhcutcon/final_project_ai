# Story 2.2: Auto Data Type Detection

## Status

Draft

## Story

**As a** user,
**I want** the system to automatically detect if my data is tabular, time-series or mixed,
**so that** the appropriate AI model is selected.

## Acceptance Criteria

1: `DataService.detect_data_type(dataset_id)` phân tích columns và trả về: `tabular`, `timeseries`, hoặc `mixed`
2: Logic detection: nếu có ≥ 1 datetime column VÀ data có sequential ordering → `timeseries`; nếu chỉ có numeric/categorical → `tabular`; nếu kết hợp → `mixed`
3: Kết quả lưu vào field `data_type` của bảng `datasets`
4: Detection chạy tự động sau khi upload thành công
5: Confidence score cho detection result (0.0 - 1.0)

## Tasks / Subtasks

- [ ] Task 1 — Define DataTypeEnum and detection schemas (AC: 1, 5)
  - [ ] Add `DataTypeEnum` to `{backend}/app/schemas/upload.py`: "tabular", "timeseries", "mixed"
  - [ ] Define `DataTypeResult` schema: data_type (DataTypeEnum), confidence (float 0.0-1.0), reasoning (str — brief explanation)

- [ ] Task 2 — Implement detection logic in DataService (AC: 1, 2, 5)
  - [ ] Add `async def detect_data_type(dataset_id: UUID) -> DataTypeResult` to `{backend}/app/services/data_service.py`
  - [ ] Download CSV from MinIO via StorageService
  - [ ] Parse with pandas and analyze columns:
    - [ ] Identify datetime columns: try `pd.to_datetime()` on each column, count successes
    - [ ] Check sequential ordering: if datetime col exists, verify rows are time-ordered (monotonic or near-monotonic)
    - [ ] Count numeric vs categorical columns
  - [ ] Classification rules:
    - [ ] `timeseries`: ≥ 1 datetime col AND sequential ordering → confidence based on ordering strength
    - [ ] `tabular`: 0 datetime cols OR datetime exists but no sequential pattern → confidence based on numeric/categorical ratio
    - [ ] `mixed`: datetime + sequential + significant non-temporal features → confidence = min(timeseries_conf, tabular_conf)
  - [ ] Return DataTypeResult with confidence score

- [ ] Task 3 — Persist detection result (AC: 3)
  - [ ] After detection, update `datasets.data_type` field via SQLAlchemy
  - [ ] Also store confidence score (add `data_type_confidence` Float column to Dataset model if not present)
  - [ ] Create Alembic migration for new column if needed

- [ ] Task 4 — Trigger detection automatically after upload (AC: 4)
  - [ ] In `DataService.upload()` (Story 2.1), call `detect_data_type(dataset_id)` after successful upload and metadata save
  - [ ] Include `data_type` and `confidence` in the upload response

- [ ] Task 5 — Write tests (AC: 1, 2, 5)
  - [ ] Create `{backend}/tests/test_data_type_detection.py`
  - [ ] Test: CSV with date column + sequential data → returns "timeseries"
  - [ ] Test: CSV with only numeric/categorical → returns "tabular"
  - [ ] Test: CSV with datetime + mixed features → returns "mixed"
  - [ ] Test: confidence score is between 0.0 and 1.0
  - [ ] Test: detection result persisted to datasets table

## Dev Notes

### Relevant Source Tree

- `{backend}/app/services/data_service.py` — add `detect_data_type()` method
- `{backend}/app/schemas/upload.py` — add `DataTypeEnum`, `DataTypeResult`
- `{backend}/app/models/dataset.py` — add `data_type_confidence` column
- `{backend}/app/services/storage.py` — use `get_file()` to download CSV from MinIO

### Patterns & Conventions

- Layered architecture: detection logic lives in `DataService`, not in the endpoint
- All service methods `async def`
- Pydantic enums for type safety (`DataTypeEnum`)
- Detection triggered inside `DataService.upload()` — single responsibility, service orchestrates

### Tech Stack

- Pandas (`pd.to_datetime()`, `is_monotonic_increasing`, dtype analysis)
- SQLAlchemy 2.x (update dataset record)
- Alembic (migration for new column)
- MinIO (download file for analysis)

### Notes from Story 2.1

- `DataService` and `StorageService` are created in Story 2.1
- `Dataset` model and `DatasetResponse` schema already exist
- MinIO file key: `{user_id}/{dataset_id}/{filename}`

### Testing

- Location: `{backend}/tests/`
- Framework: pytest + httpx `AsyncClient`
- Types: unit tests for detection logic (mock CSV data), integration test for auto-trigger after upload
- Prepare fixture CSV files: one pure tabular, one time-series, one mixed
